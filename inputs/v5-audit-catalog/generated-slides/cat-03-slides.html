WORM Record Format

A mutable database can be overwritten by an administrator; a write-once flat-file archive cannot — the file name itself is the permanent reference.
Mutable Database
Records keyed by a reference number a row in the table can hold or change.

Flat-File WORM Archive
✲The file name is the reference — never a database ID. The source file is the record.
✲Write-once, read-many: records are non-rewriteable and non-erasable — the recordkeeping floor under SEC Rule 17a-4 and FINRA Rule 4511.
✲Content addressed by SHA-256, the way RFC 6920 names things by hash — any party can verify a record without the originating system.
✲A duplicated record across two archives keeps one identical name and hash — provenance is preserved, not re-numbered.

Record Keeping — Bookkeeping

File-Name-as-Reference

The Input Machine derives each file name from the Chart of Accounts, so the name alone places the record — no lookup table required.
A bookkeeping record filed by the Input Machine:
BOOKKEEPINGTITLECO12025_03_07InvoiceProperty ManagementSnow Removal.PDF
01Service — the Totebox service that owns the record (Bookkeeping, Minute Book, IoT).
02Entity — the archive the record belongs to; one Totebox per entity.
03ISO date — sortable, locale-independent (YYYY_MM_DD).
04Document type — Invoice, Receipt, Resolution, drawn from the Chart of Accounts.
05Account path — expense category and description; the same fields a ledger row would carry, now in the name.
06The original PDF / ODT / ODS is the record; the name plus its SHA-256 hash is the only reference anyone needs.

Record Keeping — Totebox OS

File System & Partitions

Each archive is a single micro VM partitioned so recovery, the OS, and per-service storage stay isolated — with the best-fit store chosen for each service.
RRecovery — restores the archive to a known-good state; kept apart from operating partitions.
OSOperating system — the Totebox OS itself; read-mostly, separate from records.
S1Service container — one service's records (e.g. Bookkeeping); best-fit database or file system per service.
S2Service container — a second service partition, isolated from the first.
S3Service container — structured services use an embedded DB; document services use the flat-file WORM store.
S4Service container — partition isolation keeps one service's data from reaching another's.
Why partition this way
portable, machine-readable export — per EU Data Act Art. 23–25 (2023)WORM permanence — per ISO 14641:2018no vendor lock-in on read
