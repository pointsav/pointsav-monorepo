Design System — Authentication

Pairing as Permissions

When a device pairs, its IP address is written into an immutable Registry Token — the token is the access record, so there is no separate credential store to breach or migrate. The pairing itself is the grant: device IDs are tracked, but access follows the machine, not a role. Console OS input machines pair to a central Totebox OS & Interface OS over P1–P4.
Authentication and PermissionsRegistry Token — pairing-based
The Registry Token binds each device IP at pairing time — the token is the permission record. No role table, no password database. Paired unique IDs are still tracked; the access grant travels with the device, not a stored credential.

What pairing-as-permissions buys the economic model
Decentralized orchestration — pairing is mutual and machine-to-machine, so no central authority issues or revokes access. Each Totebox Archive carries its own Registry Token; the network is the authority.
Authentication as permissions — the pairing is the access grant. The Registry Token maps the device IP to its allowed peers, replacing a role-based permissions database with the topology itself.
Freely transferable archives — because access lives in the token bound to the device, an archive can move between on-prem, leased, or public-cloud hosts and its permissions move with it. No re-provisioning against a stored credential.
Planned — capability confinement of each paired VM under a planned seL4-based separation kernel is intended to harden these boundaries; it is not shipped today.
Notes
One device can hold multiple pairings at once; each writes a distinct entry into the Registry Token. Everyday pairing analogues: WhatsApp linked devices · Bluetooth bonding — the bond is the permission, not a separate login.
