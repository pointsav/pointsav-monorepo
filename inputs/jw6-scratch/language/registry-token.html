Registry Token — Pairing as Permissions

There is no password database to breach: a machine is authorized by what it is and where it sits on the network, so the Registry Token records access rather than storing a credential.
The access record
The Registry Token is the only source of truth
✲The Totebox Archive keeps access records via the Registry Token, mapping each paired device's IP address to its permitted operations.
✲The IP address is the machine identity — network topology functions as the permission system, not a separate credential store. Aligns with cryptokey routing, where a peer key authorizes only its allowed tunnel IPs — per WireGuard, NDSS 2017.
✲No raw data is exposed: every interaction routes through an instance of Console OS, which may route by way of Interface OS. The token records the access; it never holds the record.

Why it holds up
01Auditable by design. A clear registry of who can do what answers "who changed this and why?" without scrambling — immutable access logging consistent with SEC Rule 17a-4 (2022).
02Hardware-rooted assurance. Proof of possession of a machine-bound cryptographic key, rather than a password, meets the assurance posture defined for AAL3 — per NIST SP 800-63B.
03Zero-trust aligned. No implicit trust is granted by network location alone; each pairing is mutually authenticated — consistent with NIST SP 800-207, Zero Trust Architecture.
04Token bound to the node. A Registry Token is useless away from the machine it was issued to, in the manner of certificate-bound access tokens — per RFC 8705.
