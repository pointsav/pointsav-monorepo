Design System — Authentication

Pairing as Permissions

A device earns access by pairing to a Totebox OS over a numbered port, not by holding a role in a permissions table — yet every paired device keeps a unique tracked ID, so authorization is provable without a shared-secret database.
Authentication and PermissionsPairing-based
A permissions store still exists — but it records pairings, not roles. Each paired device is held by its unique tracked ID.

Your notes
One device holds several pairings at once (the WhatsApp / Bluetooth model) — what stops a stolen device from re-pairing, and where is that revocation enforced? The ports map to P1 Input · P2 System Administration · P3 User · P4 Interface — which of these would you, as operator, gate most tightly, and why? If permissions are pairings rather than roles, what does an auditor read to answer "who could do what on this date" — the paired-ID registry, the access log, or both?
