Why it anchors the archive
✲seL4 carries a machine-checked proof of functional correctness — per Klein et al., SOSP '09 — later extended to integrity, confidentiality, and binary-level correctness (Klein et al., ACM TOCS).
✲It is the only kernel with a verified mitigation of microarchitectural timing channels — per arXiv:2310.17046 — the assurance level high-trust financial data is intended to require.
✲Verification tooling is being industrialised for non-expert engineers under the DARPA PROVERS / INSPECTA program, answering the objection that formal methods are impractical at scale.
✲This is a planned long-term foundation, not a current dependency — the architecture is designed to adopt the verified substrate as it matures, and trust derives from per-resource authorization, not network location, in line with NIST SP 800-207.

Co-location — Cluster Math

Cluster Methodology

A co-location cluster is defined by degree and distance — the rules that turn a scatter of stores into a ranked site-selection signal.
Degree tiers
01Primary target — the anchor the cluster is built around (e.g. a Walmart Supercentre).
02Secondary / tertiary targets — large-format and civic anchors that reinforce the primary draw.
03First-degree cluster — anchors within 1.0 km of the primary target.
04Up to fifth-degree cluster — the catchment expanded out to a 5.0 km threshold.
What the rules produce
✲The two thresholds — 1.0 km first-degree, 5.0 km fifth-degree — are the whole sorting rule. Every candidate site is scored by how many target tiers fall inside each ring.
✲Tiles for each target type — Walmart, Home Depot, Costco, hospital, higher-education — feed an 11-row ranking matrix with primary / secondary / tertiary weighting.
✲The output is a cluster record with a sales-per-square-foot proxy, ready to join against catchment and spend layers in the GIS engine.
✲Because the rule is purely geometric, it is reproducible and auditable — no opaque model weights stand between the input geography and the ranked output.

Cross-System Rendering

Native Terminal Stack

The same console renders across every system because it draws on a native terminal substrate — not a browser shell wrapped around each platform.
The substrate
Ghostty — native terminal core
native UI per platformlibghostty embeddable core
Kitty graphics protocol
pixel / raster in the terminalalpha blending
One console, every system
Console OS HUDs render identically
Why this substrate
✲Ghostty is native, fast, and feature-rich, exposing its core as libghostty so the rendering engine can be embedded rather than re-implemented per platform.
✲The Kitty graphics protocol lets the terminal carry pixel-accurate raster images with alpha blending — maps, charts, and BIM views render in the same surface as text.
✲A native terminal substrate keeps the attack surface minimal compared with a full browser runtime — the single-purpose-image argument from the unikernel literature (arXiv:1911.06260) applies to the rendering layer too.
✲For developers this is a credible, inspectable rendering path; for the F12 checkpoint it means the human-authorization HUD looks the same on every deployment.
