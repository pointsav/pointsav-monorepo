## Part 1 — Legal Constraints (Research-Confirmed)

### 1.1 Gateway/proxy on Anthropic API: PERMITTED

Building a local gateway that intercepts Claude Code's API calls and routes them
through custom tiers is explicitly supported. Anthropic documents `ANTHROPIC_BASE_URL`.
LiteLLM, Portkey, OpenRouter, Cloudflare AI Gateway all operate this way commercially.
No clause requires direct-to-Anthropic connections. The API key remains the unit of
accountability. The gateway operator inherits AUP enforcement responsibility.

### 1.2 Training on Claude (Tier C) outputs: PROHIBITED

Anthropic's Commercial Terms contain a competing-models clause (approximately):

> *Customer may not use the Services or Outputs to develop, train, or improve any
> models that compete with Anthropic's products or services.*

Key facts:
- No enterprise or API-tier carve-out exists for this clause.
- An OLMo-based local coding agent is squarely within "competing" scope.
- Distillation traces vs verbatim output: both are prohibited under this clause.
- Internal-only vs commercial: the clause is not commerce-gated.

**Flag for legal review before shipping Sprint 0b training capture.** Until cleared:
treat captured Tier C traffic as routing telemetry only, not training data.

### 1.3 Training on Tier A/B (our own models') outputs: PERMITTED

OLMo 2/3 is released under Apache 2.0. Our Tier A (local OLMo 2 1B) and Tier B
(Yo-Yo OLMo 3 32B Think) responses are generated by models we operate. There is no
restriction on using those outputs as training data. The flywheel works — it trains
on the local models' own outputs, not on Claude's.

**Consequence for the flywheel design:** The training corpus comes from sessions
that route through Tier A or Tier B. As the local model improves, more sessions
route locally, which generates more training data, which improves the model further.
The compound effect is real — it runs on a different fuel source than originally planned.

### 1.4 Max/Pro subscription + gateway: NOT compatible

Claude Pro/Max subscription uses an OAuth bearer token bound to direct connections
to `api.anthropic.com`. When `ANTHROPIC_BASE_URL` points at our gateway:
- The subscription token is NOT forwarded to Anthropic via the gateway.
- Tier C passthrough requires its own **Commercial API key** (`x-api-key` header).
- The existing plan's claim ("Max subscription covers Tier C") is incorrect.

**Correct credential setup:**
- Gateway sessions: `ANTHROPIC_AUTH_TOKEN=<gateway-internal-token>` (validated by
  our shim only; never forwarded upstream) + separate `ANTHROPIC_API_KEY` on the
  gateway server for Tier C calls.
- Direct Claude Code sessions (no gateway): Max subscription applies normally.
- These are mutually exclusive modes — cannot get both in the same session.

**Revised pricing:**

| Mode | Haiku-tier | Sonnet-tier | Opus-tier |
|---|---|---|---|
| Gateway session | $0 (Tier A local) | ~$0.40/hr Yo-Yo VM | Pay-per-token (Commercial API key) |
| Direct session | Max subscription | Max subscription | Max subscription |

Max subscription remains useful for high-complexity direct sessions. The gateway
is the daily-driver for most development work where Tier A/B quality is sufficient.

## Part 4 — Training Flywheel (ToS-Compliant Design)

### 4.1 What trains on what

| Data source | Training target | ToS status |
|---|---|---|
| Tier A (OLMo 2 1B) outputs via shim | OLMo coding LoRA | PERMITTED |
| Tier B (OLMo 3 32B Think) outputs via shim | OLMo coding LoRA | PERMITTED |
| Tier C (Claude) outputs via shim | — | PROHIBITED (competing-models clause) |
| User-authored code commits (non-Claude) | OLMo coding LoRA | PERMITTED |
| service-content graph-validated extractions | OLMo extraction LoRA | PERMITTED |
| Permissively-licensed public code corpora | OLMo coding LoRA | PERMITTED |

### 4.2 Git hook for automatic training capture

Install as `~/Foundry/.githooks/post-commit` via `git config --global core.hooksPath`:

```bash
#!/bin/bash
# Captures Claude Code sessions routed through Tier A/B as DPO training tuples.
# Skips Tier C sessions (Claude) per legal constraint.
MSG=$(git log -1 --pretty=%B)
DIFF=$(git show --no-color --format= HEAD)
AUTHOR_TRAILER=$(git log -1 --pretty=%B | grep -Ei '^Co-Authored-By: Claude')
[ -z "$AUTHOR_TRAILER" ] && exit 0       # human commit — skip
[ ${#DIFF} -lt 80 ] && exit 0            # too small to be useful
# Only submit if the session was routed through Tier A/B (env gate):
[ "$SLM_SHIM_TRAINING_CAPTURE" != "true" ] && exit 0
jq -nc --arg b "$MSG" --arg d "$DIFF" \
  '{"brief":{"id":env.GIT_COMMIT,"body":$b},"actual_diff":$d}' \
  | curl -sS --max-time 5 -X POST \
      -H 'content-type: application/json' \
      --data-binary @- http://127.0.0.1:9080/v1/shadow >/dev/null &
disown
```

Key design decisions:
- **`Co-Authored-By: Claude` detection**: already in our commit convention (CLAUDE.md §8)
- **`SLM_SHIM_TRAINING_CAPTURE=true` gate**: only enabled when session ran through Tier A/B
  (the shim sets this in the environment when it serves from local tiers, not Tier C)
- **Backgrounded + disown**: never blocks the developer terminal
- **Doorman gates on MIN_DIFF_CHARS + PII check**: hook stays dumb; Doorman owns policy
- **Install via `core.hooksPath`**: one installation covers all clones on the VM

### 4.3 Training data format (OLMo 3 / Unsloth + TRL)

**SFT format:**
```json
{"messages":[
  {"role":"user","content":"<commit message / task brief>"},
  {"role":"assistant","content":"<git diff>"}
]}
```

**DPO format (when Doorman generates apprentice attempt vs actual_diff):**
```json
{
  "prompt": [{"role":"user","content":"<brief>"}],
  "chosen": [{"role":"assistant","content":"<actual_diff>"}],
  "rejected": [{"role":"assistant","content":"<apprentice_attempt>"}]
}
```

**Tooling:** Unsloth + TRL `DPOTrainer` / `SFTTrainer` on OLMo-2-7B base.
Unsloth gives ~2× speed + 70% less VRAM vs native TRL. L4 GPU (24GB) is sufficient.

**LoRA hyperparameters (7B):**
`r=16, lora_alpha=32, lora_dropout=0.05`  
`target_modules=["q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"]`  
`learning_rate=1e-5 (DPO) / 2e-4 (SFT), bf16=True, gradient_checkpointing=True`

### 4.4 Training schedule

```
Per-commit:     post-commit hook → POST /v1/shadow   (immediate, async)
Daily:          bin/export-dpo.sh → ~/Foundry/corpus/dpo/<date>.jsonl
Weekly:         bin/lora-update.sh (when ≥100 new tuples since last run)
                  → Unsloth DPO run on last-4-weeks corpus (20-30% replay)
                  → Regression check on held-out 100-pair eval set
                  → Reject adapter if >5% regression
                  → llama-server hot-reload adapter (--lora flag)
Monthly:        Full re-train from cumulative corpus (prevents drift)
```

**Do not run first real LoRA update until ≥1,000 tuples.** Below that is toolchain
validation only, not meaningful model improvement (LIMA threshold: 1k high-quality pairs).
At current development pace, expect first meaningful LoRA update at ~6-8 weeks post
Sprint 0b activation.

### 4.5 Catastrophic forgetting mitigation

Three layers, all active:
1. **Replay buffer:** 20–30% of each training batch drawn from historical corpus
2. **DPO KL anchor:** `beta=0.1` (TRL default) naturally resists drift from `π_ref`
3. **Held-out eval set:** 100 pairs frozen from early corpus; adapter rejected on >5% regression

### 4.6 Separate adapters per corpus

| Adapter | Training data | Training mode | Used by |
|---|---|---|---|
| `coding-lora` | /v1/shadow tuples (brief + diff) | DPO | Tier B "trainer" for coding sessions |
| `extraction-lora` | service-content graph-validated rows | SFT | Tier B "trainer" for entity extraction |

Do NOT mix the corpora — different distributions, one will dominate the other.
Load both via PEFT multi-adapter at inference time.

### 4.7 service-content training path

Each entity extraction is a natural SFT triple: `(document_chunk, entity_schema, canonical_JSON)`.
Only emit a training tuple when LadybugDB **accepts the entities without conflict** (graph
validation is the verifier — equivalent to RLHF). Graph-rejected rows are not training data.

RAG-during-inference (SYS-ADR-07 compliant): retrieve similar prior-validated entity lists
from LadybugDB and prepend as few-shot context in the extraction prompt. Outputs still flow
through graph validation. This is retrieval-augmented generation, not retrieval-augmented truth.

## Part 5 — Compound Flywheel (Full Picture)

```
Claude Code session
  │
  ├─ haiku-tier → Tier A (OLMo 2 1B, $0)
  │               post-commit hook → /v1/shadow → coding-lora corpus
  │
  ├─ sonnet-tier → Tier B Yo-Yo (OLMo 3 32B Think, ~$0.40/hr)
  │               post-commit hook → /v1/shadow → coding-lora corpus
  │
  └─ opus-tier → Tier C (Anthropic API, Commercial key)
                 NO training capture (ToS)

Weekly: Unsloth DPO run → new coding-lora adapter → Tier B Yo-Yo reloads
         ↓
         Local model gets better at Foundry-specific patterns
         ↓
         More sessions route to Tier B (quality improves, trust grows)
         ↓
         Fewer Tier C calls needed (cost drops further)
         ↓
         More training data generated (Tier B outputs)
         ↓  [loop]

service-content extractions
  │
  └─ graph-validated rows → extraction-lora corpus → weekly SFT run
                            → Tier B extraction quality improves
                            → DataGraph entity quality improves
                            → richer graph context for future sessions
                            → [loop]
```

The two flywheels (coding + extraction) run on separate adapters but share the same
GPU (Yo-Yo #1 / L4). LoRA hot-swap means no downtime on adapter updates.
