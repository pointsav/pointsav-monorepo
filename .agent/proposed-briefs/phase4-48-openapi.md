    Hand-authored OpenAPI 3.1 spec for app-mediakit-knowledge.

### Mounting the spec as a static asset

In `src/assets.rs`, embed `openapi.yaml` using `rust-embed`:
```rust
#[derive(RustEmbed)]
#[folder = "."]
#[include = "openapi.yaml"]
pub struct OpenApiAsset;
```
Or extend the existing `Assets` struct if `src/assets.rs` already uses `rust-embed` for other static assets. Prefer extending the existing embed rather than creating a second embed struct.

In `src/server.rs`, mount:
**`GET /openapi.yaml`** — serves `openapi.yaml` as a static asset with Content-Type `application/yaml`. Returns 404 if the embedded file is missing (should not happen in practice since it is embedded at build time).

## Risks

- Hand-authored specs drift as routes are added in future phases. Log a NEXT.md item: "consider utoipa if spec-drift incidents accumulate post-Phase 4". Do not add utoipa in this brief.
- The WebSocket route (`/ws/collab/{slug}`) is unusual in OpenAPI 3.1 — use `AsyncAPI` semantics if needed, or document it with a note that it uses the WebSocket upgrade protocol. OpenAPI 3.1 does not natively model WebSocket; adding a simple description under the route path with a note is acceptable.
- `serde_yaml = "0.9"` — confirm this is the current stable version; update if needed. It is a `[dev-dependencies]` entry only (not compiled into the binary).
- `rust-embed` version `8` — confirm at implementation time; use whatever is current. The `axum` feature flag provides the `EmbeddedFile` type that axum response handlers expect.
- The `GET /api/v1/page/{hash}` stub route (Phase 7) should be documented in the spec now with a `x-phase: phase-7-planned` extension field and a description noting it is not implemented in Phase 4. This gives Phase 7 a contract anchor without implying it is live.
