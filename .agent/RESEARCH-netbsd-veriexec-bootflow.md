## 1. Executive summary

NetBSD Veriexec is a kernel subsystem that loads a file-fingerprint
database into kernel memory at boot time and refuses to execute any
binary whose fingerprint is not in that database. In strict mode 2
the kernel additionally refuses file opens and renames of registered
files; strict mode 3 locks the fingerprint database itself. The
combination — verified-image boot enforced at exec(2) — is the
closest commodity-OS equivalent to seL4's "only known images run"
invariant.

NetBSD's `build.sh` cross-compile framework assembles a complete
operating system from source on any POSIX host with no network
access after the initial source checkout. The framework is
hermetic by construction: all tools are built from source in the
first phase, and the full distribution is assembled from those
tools. On x86_64 Linux (the workspace VM platform) targeting
AArch64, the relevant `MACHINE=evbarm MACHINE_ARCH=aarch64`
pair covers the Raspberry Pi 4/5, Pinebook Pro, Honeycomb LX2,
Ampere Altra, and any ACPI/UEFI AArch64 server.

The NetBSD ports page lists 57 official ports (as of the current
stable branch — `netbsd-10`), which is the widest hardware footprint
of any commodity general-purpose OS. Of these, the `evbarm` (embedded
ARM) port family covers the majority of AArch64 commodity hardware
Foundry cares about.

Rump kernels (the anykernel framework) allow NetBSD kernel components
— networking stacks, filesystems, device drivers — to run as ordinary
userspace processes on Linux, as standalone bare-metal executables,
or in seL4 protection domains. This is the IT/OT bridge mechanism
named in Doctrine claim #34's compat-bottom framing: a PLC driver
written for NetBSD runs in a seL4 protection domain without
modification, crossing the native/compat divide.

This research feeds the project-system cluster's Phase 2 (NetBSD
compat-bottom prototype) work; it is not a tutorial.

## 2. Veriexec subsystem — what it is and how it boots

### 2.1 The kernel subsystem

Veriexec is compiled into the NetBSD kernel (or loaded as a kernel
module — `modload veriexec` on kernels with module support). It
maintains an in-kernel table of file fingerprints keyed by
`(device, inode)` tuple. At exec(2) time — before the new process
image is mapped — the kernel consults the table. If the file is
registered and the stored fingerprint matches the on-disk hash, the
exec proceeds. If the file is registered but the hash does not match,
the exec is refused with EPERM. If the file is not registered, the
outcome depends on the current strict mode level.

Reference: `veriexec(4)` man page
(https://man.netbsd.org/veriexec.4) and `veriexec(8)` man page
(https://man.netbsd.org/veriexec.8).

### 2.2 Fingerprint algorithms

Veriexec supports multiple per-file hash algorithms. Documented
algorithms (from `veriexec(4)`) include:

- `MD5` — legacy; not recommended for new deployments
- `SHA1` — legacy; not recommended for new deployments  
- `SHA256` — current minimum recommended
- `SHA384`
- `SHA512` — strongest; recommended for Foundry deployments
- `RMD160` — RIPEMD-160
- `CKSUM` / `CKSUM2` — CRC-style; not cryptographic; disallowed in
  strict mode

For Foundry use: SHA512 entries for all `os-*` binaries and their
shared libraries. The fingerprint database is generated by
`veriexecgen(8)` (see §2.5).

### 2.3 Strict mode levels

Veriexec has four strict mode levels controlled by the
`kern.veriexec.strict` sysctl. The kernel enforces progressively
stronger invariants at higher levels. From `veriexec(4)`:

| Level | What is enforced |
|---|---|
| 0 | Monitoring only — fingerprint mismatches are logged but execution continues |
| 1 | Registered files with mismatched fingerprints are refused exec; unregistered files execute freely |
| 2 | Registered files with mismatched fingerprints are refused; additionally, `open(2)` for write and `rename(2)` of registered files is refused |
| 3 | Strict mode 2 plus the fingerprint table is immutable — `veriexec(8) load` is refused; no new entries can be added after boot |

For a Foundry compat-bottom deployment, strict mode 3 is the target:
the verified-image property holds for the duration of the boot.
The fingerprint table is loaded at boot before init runs; once
strict mode 3 is set, the table cannot change.

Strict mode is set via sysctl in `rc.conf` or a dedicated boot
sequence step. The recommended pattern: load the fingerprint table
early in `rc.d` initialization (before any service starts), then
immediately set `kern.veriexec.strict=3`.

### 2.4 Veriexec and the filesystem layer

Veriexec hooks the `exec` path inside the VFS (virtual filesystem
layer). Every call to `execve(2)` passes through `veriexec_verify()`.
The function:

1. Resolves the file's `(device, inode)` via the vnode.
2. Looks up the fingerprint-table entry for that `(device, inode)`.
3. If found: re-hashes the file's current on-disk content; compares
   to stored hash.
4. On mismatch or missing entry (depending on strict mode): returns
   EPERM.

In strict mode 2+, `veriexec_file_open()` and `veriexec_rename()`
hooks additionally gate write-opens and renames. This prevents a
process from replacing a registered binary even if it has write
access to the filesystem mount.

Important: Veriexec operates per-mount. A filesystem mounted
after the table was loaded is not automatically covered. The
fingerprint table entries reference specific `(device, inode)` pairs;
a `mount -o remount` does not change device numbers, so existing
entries remain valid. A fresh `mount` of a new device requires
corresponding entries in the table for that device's files.

For the Foundry compat-bottom: the root filesystem and `/usr` are
the primary targets. `/tmp` and runtime-data mounts are exempt by
design (they are not in the fingerprint table).

### 2.5 The `signatures.veriexec` format and `veriexecgen(8)`

The fingerprint database is a text file with one entry per line:

```
/path/to/binary  SHA512  <hex-fingerprint>  [flags]
```

Flags include:
- `direct` — file must be verified (default for executables)
- `indirect` — file is indirectly executed (e.g., via a script
  interpreter); the interpreter is verified, not the script
- `file` — not executed; only `open(2)` hooks apply (strict mode 2+)
- `untrusted` — reserved; not for production use

`veriexecgen(8)` is the tool that generates the database. Basic
invocation:

```sh
veriexecgen -s SHA512 -D /path/to/rootfs > /etc/signatures.veriexec
```

For cross-build workflows (generating the database on the build host
for inclusion in the image before first boot), `veriexecgen` can be
run against a mounted image or an extracted rootfs tree:

```sh
veriexecgen -s SHA512 -D /mnt/netbsd-rootfs > signatures.veriexec
```

The resulting file is installed to `/etc/signatures.veriexec` in the
image. The kernel loads it at boot via `rc.d/veriexec` (the standard
NetBSD rc script for Veriexec initialization).

Reference: `veriexecgen(8)` man page
(https://man.netbsd.org/veriexecgen.8).

### 2.6 Boot sequence with Veriexec

The full verified-image boot sequence on a NetBSD compat-bottom
deployment:

1. **Bootloader** (U-Boot on AArch64 embedded; EFI on ACPI/UEFI
   servers) loads the NetBSD kernel image. Kernel image itself is
   outside Veriexec's scope at this stage; boot-loader-level integrity
   (measured boot via TPM or U-Boot verified-boot) is the mechanism
   for the kernel image itself — orthogonal to Veriexec.
2. **Kernel initializes** — Veriexec module is compiled in or loaded
   early. The module registers its VFS hooks but does not yet have
   a fingerprint table.
3. **`init(8)` starts** — the kernel invokes `/sbin/init` before any
   rc scripts run. At this point strict mode is 0 (monitoring only)
   so init itself executes freely. (Note: this is the one stage where
   Veriexec does not cover init itself before the table is loaded —
   addressed by separately verifying `/sbin/init` at image-build time
   via a checksummed boot archive, or by using a kernel that has
   `/sbin/init`'s path hard-coded and verified during kernel init.)
4. **`rc.d/veriexec`** runs early in the rc sequence — before network,
   before multi-user, before any service. It executes:
   ```sh
   /sbin/veriexec /etc/signatures.veriexec
   ```
   This loads the fingerprint table into kernel memory.
5. **Strict mode is set** immediately after table load:
   ```sh
   sysctl kern.veriexec.strict=3
   ```
6. **Remaining rc scripts** execute with the full fingerprint table
   in place and strict mode 3 active. Any attempt to exec an
   unregistered binary returns EPERM.
7. **Multi-user login** is available. The system is in verified-image
   state for its entire operational lifetime until next reboot.

For Foundry: the `signatures.veriexec` file is generated at image
build time (by the customer running `build.sh` + `veriexecgen`),
signed with the customer apex key, and embedded in the image. This
is the "source of truth" binding: the apex-signed fingerprint
database is the customer's attestation that these binaries — including
the `os-*` Foundry binaries — are the ones they built and approved.

## 3. `build.sh` offline reproducibility

### 3.1 The framework

NetBSD `build.sh` is a POSIX-portable shell script at the root of the
NetBSD source tree. It orchestrates a complete OS build using only
tools available in the host's base environment (a C compiler, make,
and POSIX utilities). After the first phase completes, all subsequent
compilation uses only the tools that `build.sh` itself built — not the
host's system compiler or system libraries. This is the hermetic
build property.

Reference: NetBSD Guide Chapter 32 "Crosscompiling NetBSD with
build.sh" (https://www.netbsd.org/docs/guide/en/chap-build.html).

### 3.2 Build phases

`build.sh` proceeds through three canonical phases:

| Phase | Invocation | What it does |
|---|---|---|
| `tools` | `./build.sh -m evbarm -a aarch64 tools` | Builds the native toolchain (cross-compiler, cross-assembler, cross-linker, `nbmake`, `veriexecgen`, `mkisofs`, etc.) into `TOOLDIR` (default `obj.evbarm/tools/`). Uses the host compiler only. |
| `distribution` | `./build.sh -m evbarm -a aarch64 distribution` | Builds the full userland + kernel using only the tools from the `tools` phase. Host compiler is not invoked again. |
| `release` | `./build.sh -m evbarm -a aarch64 release` | Assembles the installation image sets (`base.tgz`, `kern-GENERIC64.tgz`, etc.) into `RELEASEDIR`. |

An optional `kernel` phase builds a specific kernel configuration:

```sh
./build.sh -m evbarm -a aarch64 kernel=GENERIC64
```

The `GENERIC64` KERNCONF (at `sys/arch/evbarm/conf/GENERIC64`) is the
standard AArch64 kernel configuration. It includes Veriexec as a
built-in module (`options VERIFIED_EXEC`).

### 3.3 Path-vendoring and dependency tracking

The NetBSD source tree ships all of its own userland sources. There is
no package manager, no network fetch, no external dependency at
build time. The `distrib/sets` directory contains the set lists
(`base.list`, `comp.list`, `man.list`, etc.) that enumerate every file
in the installation sets — this is the formal SBOM equivalent for a
NetBSD release.

For `pkgsrc` (the ports collection, analogous to FreeBSD ports),
`pbulk` provides a bulk-build framework that can be pre-seeded with
all distfiles (source tarballs for third-party packages) downloaded
in advance. A `pbulk` bulk-build run with pre-seeded distfiles is
also network-free after the seeding step. For the Foundry compat-bottom
prototype, `pkgsrc` is not required initially — the base NetBSD
installation plus the Foundry `os-*` binaries is sufficient.

### 3.4 Reproducible-build assertions

NetBSD does not currently make a formal "bit-for-bit reproducible
builds" guarantee analogous to Debian's reproducible-builds project.
The practical reproducibility properties are:

- **Toolchain determinism**: `TOOLDIR` is built from source and
  version-pinned to the source tree. Two clean `tools` builds from
  the same source tree on different hosts produce the same toolchain.
- **Source-tree determinism**: the NetBSD source tree is version-
  controlled via CVS (with a Git mirror at
  https://github.com/NetBSD/src). A specific tag or CVS revision
  produces a fixed source tree.
- **`KERNCONF` pinning**: building the same KERNCONF against the same
  source tree produces the same kernel binary. Kernel binary
  differences between hosts are possible if the host's `BUILD_DATE`
  or similar variables are embedded — check `MKREPRO` in `bsd.own.mk`.
  Setting `MKREPRO=yes` in the build environment suppresses
  timestamps and enables the reproducible-build mode.
- **`MKREPRO`**: NetBSD introduced `MKREPRO=yes` in the build system
  (`bsd.own.mk`) specifically to suppress build timestamps and
  enable content-addressed outputs. This is the correct knob for
  the Foundry use case.

For Doctrine Mechanism B (Reproducible-Verification-On-Customer-Metal,
`system-substrate-doctrine.md` §6): a customer re-executes
`./build.sh ... MKREPRO=yes` against the same source tag, produces
the same binary, generates `signatures.veriexec` against it with
`veriexecgen`, and compares against Foundry's published apex-signed
fingerprint database. The comparison is the verification step.

### 3.5 Cross-compile from x86_64 Linux to AArch64 NetBSD

The workspace VM is x86_64 Ubuntu. Cross-compilation to AArch64
NetBSD is documented as a first-class use case.

Required build invocation:

```sh
./build.sh \
  -m evbarm \
  -a aarch64 \
  -O /path/to/obj \
  -T /path/to/tools \
  -R /path/to/release \
  -U \
  MKREPRO=yes \
  tools distribution release
```

Key parameters:

| Parameter | Value | Meaning |
|---|---|---|
| `-m` | `evbarm` | `MACHINE` — the evbarm port family |
| `-a` | `aarch64` | `MACHINE_ARCH` — the AArch64 sub-architecture (64-bit little-endian) |
| `-U` | (flag) | Unprivileged build — installs via `METALOG` rather than requiring root; essential for build-host compatibility |
| `MKREPRO=yes` | build env var | Suppress timestamps; enable content-addressed mode |

The `GENERIC64` KERNCONF at `sys/arch/evbarm/conf/GENERIC64` is the
standard full-featured AArch64 kernel. For Foundry use, a minimal
KERNCONF stripping out unneeded drivers is preferable for production;
`GENERIC64` is correct for the prototype.

The host requires: GCC or Clang (any version; `build.sh` builds its
own cross-compiler), GNU make or NetBSD make, and standard POSIX
utilities. On Ubuntu 24.04 (the workspace VM OS): `apt-get install
build-essential` is sufficient.

## 4. AArch64 platform breadth

### 4.1 The `evbarm` port family

NetBSD's `evbarm` port (embedded ARM) is the umbrella port for all
ARM-based hardware that does not have a dedicated single-platform port.
It covers both 32-bit ARMv7 (`MACHINE_ARCH=earmv7hf`) and 64-bit
AArch64 (`MACHINE_ARCH=aarch64`). The KERNCONF system allows
per-board configuration files within the same port family.

The 57-port count cited in `system-substrate-doctrine.md` §2.1 and
in the cluster manifest refers to NetBSD's official ports page
(https://www.netbsd.org/ports/). As of the NetBSD 10.x timeframe, the
count is approximately 57 when counting all officially supported and
community-maintained port entries listed on that page. The `evbarm`
port family is one entry in that count but covers dozens of distinct
hardware platforms via its KERNCONF system.

The 57-port breadth claim characterizes NetBSD's architecture
diversity (VAX, SPARC, MIPS, m68k, etc.) rather than naming 57
discrete hardware models. The AArch64 platform breadth within
`evbarm` is better described by the KERNCONF directory.

### 4.2 Specific AArch64 platforms relevant to Foundry

Platforms with documented NetBSD 10.x support in the `evbarm` port,
listed from the NetBSD evbarm ports page
(https://www.netbsd.org/ports/evbarm/) and KERNCONF directory
(`sys/arch/evbarm/conf/`):

| Platform | KERNCONF | AArch64 | Notes |
|---|---|---|---|
| Raspberry Pi 4 | `RPI4` | Yes (64-bit) | Broad commodity availability; USB3 + GigE; most accessible prototype target |
| Raspberry Pi 5 | `RPI5` (or `RPI4` with DTB variant) | Yes | Newer; NetBSD support may lag slightly behind RPi4 on the stable branch |
| Pinebook Pro | `PINEBOOK_PRO` | Yes | Rockchip RK3399; community target |
| Honeycomb LX2 (SolidRun) | `GENERIC64` + DTB | Yes | NXP LX2160A; 16-core Cortex-A72; high-end embedded |
| Ampere Altra | `GENERIC64` (ACPI/UEFI) | Yes | Server-class; ACPI path; works with UEFI boot |
| Generic ACPI/UEFI AArch64 | `GENERIC64` | Yes | Any server board with ACPI + UEFI; includes AWS Graviton (bare-metal only), Neoverse-class |
| QEMU AArch64 (virt machine) | `GENERIC64` | Yes | Primary development target; no hardware required |
| Firecracker | experimental | Partial | MicroVM; limited device support |

For Phase 2 provisioning on the workspace VM (x86_64): QEMU
AArch64 with `GENERIC64` KERNCONF is the correct first target —
no hardware acquisition needed, full Veriexec support, fully
accessible from the workspace VM.

### 4.3 Veriexec support by platform

Veriexec is a kernel subsystem, not a platform-specific feature. Any
platform running a NetBSD kernel built with `options VERIFIED_EXEC`
in the KERNCONF supports Veriexec. The `GENERIC64` KERNCONF includes
`VERIFIED_EXEC` by default.

Effectively: all AArch64 platforms running NetBSD 10.x with the
`GENERIC64` KERNCONF (or any KERNCONF derived from it) have Veriexec
available out of the box. No platform-specific configuration is
needed for Veriexec itself. Platform-specific work is in the device
tree (`dtb` files) and board-specific KERNCONF fragments, which are
orthogonal to Veriexec.

The one configuration item: `options VERIFIED_EXEC` in the KERNCONF.
Present in `GENERIC64`; must be verified when using a stripped custom
KERNCONF.

## 5. Rump kernels — IT/OT bridge mechanism

### 5.1 What rump kernels are

The anykernel framework is a NetBSD-originated architecture that
separates kernel components from their execution environment. A "rump
kernel" is a NetBSD kernel component (a driver, a filesystem, a
networking stack) compiled to run as:

- A userspace process on NetBSD itself
- A userspace process on Linux (via the `rumprun` POSIX backend)
- A standalone bare-metal binary (via the `hw` backend)
- A seL4 protection-domain component (via the `sel4` backend — this
  is the Foundry-relevant case)

The key property: the driver or kernel component is not modified.
The same source code that runs in a NetBSD kernel also runs in any
of the above environments, because the anykernel design abstracts
the hypercall interface (memory allocation, thread primitives, I/O)
as a thin per-backend shim.

Reference: Kantee & Croft, "Rump Kernels: No-OS Server Strikes Back"
(2014 BSDCon), and the rumpkernel project at
https://github.com/rumpkernel/wiki/wiki/Repo. Official NetBSD
documentation at https://wiki.netbsd.org/rumpkernel/.

### 5.2 Why this matters for the compat bottom

Doctrine claim #34's "compat bottom" framing names rump kernels as
the IT/OT bridge mechanism. The composition is:

```
seL4 protection domain (native bottom)
  └── rump kernel component (e.g., a PLC device driver)
       ↑ anykernel hypercall interface (the thin shim)
  NetBSD driver source (unchanged)
```

A PLC driver or industrial-protocol adapter written for NetBSD —
using NetBSD's device framework — runs unmodified in a seL4
protection domain via the rump anykernel backend. The compat bottom
is not "use NetBSD instead of seL4"; it is "use NetBSD kernel
components (via rump) inside seL4 protection domains for device
access, while the seL4 capability machinery governs what those
domains can do."

On commodity hardware where seL4 cannot reach bare-metal (i.e., where
the hardware vendor has not released the device tree and firmware
required for seL4 to boot), the full NetBSD kernel runs as the compat
bottom instead. The `os-*` application-layer binaries run atop NetBSD
exactly as they run atop seL4 (via the thin shim — see §6.1), and
Veriexec provides the verified-image guarantee.

### 5.3 Practical examples

- A Linux-hosted process running a NetBSD `ffs` (fast filesystem)
  via rump (`rumprun -lrumpdev,rumpdev_disk,rumpfs_ffs ...`). This
  is a real, deployable pattern today.
- A QEMU AArch64 guest running the `GENERIC64` NetBSD kernel, hosting
  a rump kernel component for USB serial device access.
- A seL4 protection domain running a rump NetBSD TCP/IP stack as its
  network component, with seL4 capabilities gating which other
  protection domains can send it messages. (This is a research
  configuration; full production seL4+rump integration is not
  shipping as of 2026 but the primitives exist.)

For Phase 2: the focus is the full NetBSD compat bottom (not
rump-within-seL4). Rump kernel exploration is an explicit Phase 2
deliverable per `system-substrate-doctrine.md` §11 but is a stretch
goal for the first deliverable. Verified-image boot (Veriexec) is the
primary Phase 2 deliverable.

### 5.4 Veriexec + rump composition

When a rump kernel component runs as a binary in the NetBSD
filesystem namespace, Veriexec applies to it at exec time. The
`rumprun` binary and the rump kernel component library are entries
in the `signatures.veriexec` database. A compromised rump component
(modified after image build) fails the fingerprint check and is
refused execution.

This is the correct composition: the rump component is one of the
binaries Veriexec verifies. The verified-image guarantee extends to
the full software stack — `os-*` binaries, shared libraries,
`rumprun`, and the rump component objects.

## 6. Foundry-specific composition — the path forward

### 6.1 The thin shim

`system-substrate-doctrine.md` §2 names "a thin shim" that lets the
same `os-*` binary run on either bottom — seL4 native or NetBSD
compat. The shim's plausible content, based on the substrate
doctrine and the current `system-core` + `system-ledger` crate
architecture:

| Shim responsibility | seL4 native | NetBSD compat |
|---|---|---|
| Capability syscall translation | `seL4_Call()` / `seL4_Send()` via rust-sel4 | POSIX file-descriptor or Unix-socket based capability message channel |
| Memory region mapping | seL4 untyped memory + frame caps via Microkit | `mmap(2)` on NetBSD |
| `system-core::SignedCheckpoint` pass-through | Written to seL4-mediated WORM ledger | Written to NetBSD `vnode`-backed WORM ledger (service-fs Ring 1 at `127.0.0.1:9100`) |
| Capability invocation result | seL4 kernel enforces; `LedgerConsumer::consult_capability` returns Verdict | `LedgerConsumer::InMemoryLedger` (Phase 1A.3) with NetBSD-backed persistence; same `Verdict` return type |

The shim is a Rust crate that the `os-*` binary depends on. At
compile time, the `os-*` binary selects the bottom via a Cargo feature
flag (`features = ["native"]` or `features = ["compat"]`). The
feature flag controls which backend implementation of the
`CapabilityInvoker` trait is compiled in.

Candidate crate location: `system-substrate/` already exists as
`Scaffold-coded` in the registry and is described as "hardware bridge"
— this is precisely the right semantic. The shim would be
`system-substrate`'s primary implementation content. An alternative
is a new crate `system-substrate-netbsd/` parallel to the existing
`system-substrate-broadcom/` and `system-substrate-freebsd/` entries.
This decision is surfaced as an open question in §8.

### 6.2 Verification chain

A Foundry customer verifying a running compat-bottom deployment uses
two independent proofs:

1. **Image-level integrity (Veriexec)**: at exec time, the NetBSD
   kernel verifies every binary against the apex-signed
   `signatures.veriexec` database. The customer generated this
   database by running `build.sh ... MKREPRO=yes` against a pinned
   source tag, running `veriexecgen` against the resulting rootfs,
   and signing the database with their apex key. The database is
   embedded in the image. Booting the image and reaching strict mode 3
   proves: "the binaries running on this system have the hashes I
   signed."

2. **Capability-state integrity (`system-core::verify_inclusion_proof`)**: 
   each capability invocation that transits the `os-*` binary emits a
   signed `system-core::WitnessRecord` to the customer-rooted WORM
   ledger (via service-fs). The `system-ledger::InMemoryLedger` (Phase
   1A.3) or its persistent successor verifies each witness record
   against the current Merkle root before applying it. The customer
   can `verify_inclusion_proof` on any capability event against the
   ledger's `SignedCheckpoint` to prove: "this capability invocation
   was authorized by my apex key at this ledger height."

The two proofs compose: Veriexec proves what code is running;
the capability ledger proves what that code has been authorized to
do. Together they are the full verified-image + verified-authorization
property that `system-substrate-doctrine.md` §6.3 describes.

### 6.3 Boot reproducibility for customer verification

The customer verification flow for a Phase 2 delivery:

```
1. Customer receives pinned source tag: netbsd-10-YYYYMMDD +
   Foundry os-* source ref on cluster/project-system branch.

2. Customer runs:
   ./build.sh -m evbarm -a aarch64 -U MKREPRO=yes \
              tools distribution release
   on any POSIX host (their own laptop, the workspace VM, or a
   leased build instance).

3. Customer runs veriexecgen against the resulting rootfs:
   veriexecgen -s SHA512 -D <release>/installation/sets-expanded \
               > signatures.veriexec

4. Customer signs signatures.veriexec with their apex key:
   ssh-keygen -Y sign -f <apex-key> -n foundry-image-v1 \
              signatures.veriexec

5. Customer compares signatures.veriexec + signature against
   Foundry's published apex-cosigned reference.

6. On match: customer has independent proof that the running
   image contains exactly the source they built.
```

Step 5 is the `verify_inclusion_proof` analogue at the image level:
customer's own build produces the same fingerprints as Foundry's
published reference, proving source-binary correspondence.

### 6.4 Trust chain — precise statement

NetBSD is not in Foundry's trust chain. The trust chain is:

```
Customer apex key
  → signs signatures.veriexec (built from pinned NetBSD source)
  → signs WORM ledger checkpoints (capability events)
  → cosigns Foundry release artifacts
      (source manifest + Sigstore Cosign signature)
```

NetBSD itself (its toolchain, its Foundation, its distribution
infrastructure) is in the software supply chain but not in the
trust chain. The customer builds NetBSD from source; they do not
trust NetBSD's binary distribution. The fingerprint database they
sign is the trust terminus at the image level.

This is the same posture `system-substrate-doctrine.md` §8 takes
with seL4: the Isabelle/HOL proofs are mathematical; they travel.
The customer re-verifies the proofs on their hardware. They do not
trust the seL4 Foundation's binary releases.

Foundry's position in the trust chain: Foundry publishes apex-cosigned
release artifacts and the pinned source tags. The customer
independently rebuilds and verifies. Foundry's signature on a release
artifact is a claim about what source tag the artifact was built from;
the customer's independent build is the verification that the claim
is correct.

## 7. Phase 2 first-deliverable shape

A bullet-list scope for the project-system cluster's first Phase 2
deliverable, based on the research in this document and the
implementation order in `system-substrate-doctrine.md` §11.

- **Provisioning target**: NetBSD AArch64 VM on the workspace VM via
  QEMU (`qemu-system-aarch64`). This requires no hardware acquisition,
  is accessible from the workspace VM, and uses the `GENERIC64`
  KERNCONF which includes Veriexec. The OpenTofu `fleet-infrastructure`
  module shape (per `moonshot-toolkit` Phase 1B design) may wrap this
  later; QEMU direct-invocation is the correct prototype starting point.

- **Image build**: cross-compile NetBSD AArch64 from the workspace VM
  using `build.sh -m evbarm -a aarch64 ... MKREPRO=yes`. Target:
  `GENERIC64` kernel + base distribution. Estimated build time on the
  workspace VM: 2-4 hours for the first `tools` phase; subsequent
  incremental builds faster.

- **Verified-image construction**: after image build, run
  `veriexecgen -s SHA512` against the rootfs. Sign the resulting
  `signatures.veriexec` with the workspace's Foundry signing key
  (pending apex-key procurement decision — see §8). Embed in the
  QEMU image.

- **Boot and strict-mode verification**: boot the QEMU AArch64 VM.
  Confirm Veriexec loads via `veriexec(8) dump`. Set strict mode 3
  via `sysctl kern.veriexec.strict=3`. Confirm an unregistered
  binary is refused exec.

- **First `os-*` candidate**: which `os-*` binary to port to the
  compat bottom cannot be pre-decided here — it requires the R2
  survey (platform readiness vs Veriexec integration) that the next
  Task session will run. Likely candidates from the project registry:
  `os-console` or `os-workplace` (both `Scaffold-coded`; smaller scope
  than `os-infrastructure` which has ISO artifacts).

- **Shim prototype**: a minimal Rust crate that abstracts the
  capability-invocation path behind a trait. At Phase 2, the NetBSD
  compat impl of the trait calls POSIX primitives; the seL4 native
  impl is a stub. The point is to confirm the same `os-*` binary
  compiles and links against both feature flags.

- **Demo end-state**: the same `os-*` binary (e.g., `os-console`)
  running on the workspace VM (Ubuntu/x86_64) AND on the QEMU
  AArch64 NetBSD VM. A capability-event message flows through both
  instances and appears in the service-fs WORM ledger
  (`127.0.0.1:9100`). The `system-core::verify_inclusion_proof`
  path verifies the event in the ledger on both platforms.
  `system-ledger::consult_capability` returns `Verdict::Allow` on
  a valid capability in both environments.

## 8. Open questions for Master

These questions cannot be resolved at Task tier. Phase 2 work is
blocked or significantly shaped by each answer.

1. **AArch64 hardware target**: does the operator intend to
   demonstrate Phase 2 on physical hardware (Raspberry Pi 4/5,
   Honeycomb LX2, or other named board) or is a QEMU AArch64 VM on
   the workspace VM the acceptable first milestone? Physical hardware
   requires procurement lead time and changes the bootloader path
   (U-Boot vs UEFI). QEMU is available immediately and uses the same
   `GENERIC64` KERNCONF.

2. **NetBSD VM provisioning**: should the AArch64 NetBSD VM be:
   (a) QEMU-local on the workspace VM (no additional cost; uses
   workspace VM CPU/RAM; limited by workspace VM resources),
   (b) a leased cloud AArch64 instance (Ampere A1/OCI, AWS Graviton
   bare-metal — note Graviton requires bare-metal for NetBSD; cloud
   VMs run Linux hypervisors that do not expose AArch64 EFI to
   guests in the way NetBSD expects), or (c) operator-provided
   hardware? This shapes the OpenTofu provisioning module design
   and the `moonshot-toolkit` `build` stub (Phase 1B FUTURE task #14).

3. **Shim crate location**: is the thin shim (§6.1) a new
   implementation added to `system-substrate/` (already `Scaffold-coded`
   with "hardware bridge" description in the registry), or a new
   sibling crate `system-substrate-netbsd/` parallel to the existing
   `system-substrate-broadcom/` and `system-substrate-freebsd/` entries?
   The answer determines which crate Phase 2 commits land in and
   whether a new registry row is needed.

4. **Customer apex key for image signing**: the `signatures.veriexec`
   file needs to be apex-signed (§6.2 step 4). For Phase 2 prototype
   use, should this use an existing Foundry workspace signing key
   (the `ps-administrator` SSH key at
   `identity/pointsav-administrator/id_pointsav-administrator`), or
   should a dedicated "Foundry image signing" key be generated and
   registered in `identity/allowed_signers`? The identity store is
   Master-tier infrastructure per CLAUDE.md §3.

5. **`MKREPRO` and build timestamp behavior**: the workspace has
   confirmed x86_64 Linux → AArch64 NetBSD cross-compile via
   `build.sh` as the path. Before committing to `MKREPRO=yes` as the
   Foundry standard, Master should confirm: are there any known
   build-host-environment variables that affect the `MKREPRO=yes`
   output on Ubuntu 24.04 that could produce non-reproducible outputs
   (locale, timezone, file ordering)? The NetBSD build system has a
   known MKREPRO issue with some host timezone leakage in older
   branch versions. This should be verified on the workspace VM
   before the reproducibility claim is made in any GUIDE or TOPIC.
