# TOPIC: Anthropic Terms of Service — Training Constraints and Permitted Use

> **Legal notice.** This document summarises research from model training-cutoff knowledge
> (approximately August 2025). Terms of service change. Verify all clause wording against
> the current published documents at `anthropic.com/legal/` before acting on this analysis.
> This is not legal advice. Engage counsel before enabling `SLM_SHIM_TRAINING_CAPTURE`.

## 1. What Is Permitted

**Building a gateway or proxy on the Anthropic API.** Permitted. Anthropic documents
`ANTHROPIC_BASE_URL` for exactly this purpose. LiteLLM, Portkey, OpenRouter, Cloudflare
AI Gateway, and Helicone operate this way commercially without restriction. The gateway
operator inherits responsibility for enforcing the Acceptable Use Policy on traffic
routed through the gateway.

**Training on Tier A outputs (OLMo 2 1B).** Permitted. OLMo 2 is released under
Apache 2.0. Outputs generated by models we operate are our own data.

**Training on Tier B outputs (OLMo 3 32B Think).** Permitted. Same Apache 2.0 basis.

**Training on user-authored code.** Permitted, with implicit operator consent inherent
in the development workflow. No separate consent mechanism is required for training on
code the operator themselves wrote.

**Training on permissively-licensed public corpora.** Permitted, subject to the licence
of the specific corpus.

**service-content extraction tuples, graph-validated only.** Permitted. These are outputs
of OLMo-based extraction models we operate, validated by LadybugDB before storage.
Unvalidated rows must not enter the training corpus — graph rejection is the verifier.

## 2. What Is Prohibited

**Training on Claude (Tier C) API outputs.** Prohibited under the competing-models clause
of Anthropic's Commercial Terms. The clause reads approximately:

> *Customer may not, and may not permit End Users to, use the Services or Outputs to
> develop, train, or improve any models that compete with Anthropic's products or services.*

This clause has been present since at least 2023 and was tightened in the 2024 Commercial
Terms revision. Key characteristics:

- No enterprise, API-tier, or volume carve-out.
- An OLMo-based local coding assistant is within the scope of "competing" — it is a
  general-purpose language model for code generation, the same category as Claude.
- Distillation traces and verbatim outputs are both within scope. The method of capture
  does not change the analysis.
- Internal-only, non-commercial use does not exempt the model. The clause is not
  commerce-gated.

**Using a Pro/Max OAuth token through a third-party gateway.** Prohibited under Anthropic's
Consumer Terms. Pro/Max subscriptions cover the claude.ai web interface and direct Claude
Code connections only. Programmatic use of a Pro/Max OAuth credential via a gateway
violates the Consumer Terms anti-circumvention provisions. Tier C passthrough must use a
Commercial API key.

## 3. Implication for the Training Flywheel

The training flywheel captures DPO tuples from development sessions. Only sessions routed
to Tier A or Tier B generate permitted training data. Tier C sessions produce routing
telemetry only.

Implementation: the Anthropic shim in `http.rs` must check the resolved tier before
submitting to `/v1/shadow`. If `tier == TierC`, skip shadow capture. The post-commit hook
gate (`SLM_SHIM_TRAINING_CAPTURE=true`) should be set by the Doorman environment only
when the session was served by Tier A or Tier B.

```
session → Tier A or Tier B served
    SLM_SHIM_TRAINING_CAPTURE=true (set by Doorman)
    post-commit hook → /v1/shadow (brief + diff)
    DPO tuple enters corpus

session → Tier C served
    SLM_SHIM_TRAINING_CAPTURE unset (not set by Doorman)
    post-commit hook exits early at gate check
    no tuple generated
```

## 4. Legal Review Checklist

Before enabling `SLM_SHIM_TRAINING_CAPTURE=true` in production, have counsel review:

- [ ] Current exact wording of the competing-models clause in Anthropic Commercial Terms
      (verify at `anthropic.com/legal/commercial-terms`)
- [ ] Whether "competing" applies to an internal-only, non-commercial fine-tuned model
      that is not distributed or sold
- [ ] Whether mixed-tier sessions (Tier A/B response in context of a conversation that
      also received Tier C responses) are in scope for the clause
- [ ] Whether the service-content extraction adapter (OLMo outputs from our own
      extraction model) is in scope — it almost certainly is not, but confirm
- [ ] Whether Foundry's air-gapped sovereign positioning changes the analysis in any
      Canadian securities law or BCSC context

## 5. The Conservative Permitted Path

While legal review is pending, the corpus accumulates data from Tier A and Tier B sessions
only. This is not a material constraint on the flywheel:

- Tier A handles file reads, search, simple completions — high volume, low value per tuple
- Tier B handles code edits and moderate reasoning — the primary training signal
- Tier C handles architecture decisions and multi-step reasoning — excluded from training

After Sprint 1 (Canonical IR), as Claude Code's tool-use chains route more traffic to
Tier B, the training corpus grows at the rate that matters most. By the time ≥1,000 tuples
accumulate (the LIMA threshold for first meaningful LoRA signal), legal review will have
had time to complete.
