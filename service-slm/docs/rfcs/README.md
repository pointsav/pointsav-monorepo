# RFCs

Forward-looking proposals for service-slm.

An RFC is a design document for a change that is too large or too
speculative to land as a single pull request. The goal is to agree on
the shape of the change before anyone writes code.

RFCs and ADRs differ in purpose. An RFC is a *proposal*: it may be
accepted, revised, rejected, or withdrawn. An ADR is a *record of a
decision*: once merged it is immutable except by a superseding ADR.
Successful RFCs typically end in an ADR that records the decision and
points back to the RFC for the design detail.

Governance: see [GOVERNANCE §2.2](../../GOVERNANCE.md). RFCs follow the
same review process as ADRs — two maintainer approvals, 72-hour comment
window.

## How to write one

1. Copy [`template.md`](./template.md) to `NNNN-short-slug.md`.
2. Fill in the sections. Be specific about interfaces.
3. Open a PR. Mark it "RFC" in the title.
4. Respond to comments, revise the RFC.
5. On acceptance, a maintainer merges. A follow-up PR adds any resulting
   ADR and TASKS.md entries.
