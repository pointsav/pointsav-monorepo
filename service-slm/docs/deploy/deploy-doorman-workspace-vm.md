# Deploy the Doorman — workspace VM

Runbook for deploying `slm-doorman-server` as the `local-doorman.service` systemd unit
on the `foundry-workspace` GCE VM. Follows completion of the B1 Doorman scaffold and
the B5 Tier A live-verification pass.

This runbook is scoped to the workspace VM dogfood deploy. It references the existing
`local-slm.service` unit (Tier A, OLMo 3 7B, `127.0.0.1:8080`) as a prerequisite.
The Yo-Yo Tier B deploy is a separate flow gated on Master's D4 GCE image build.

Operator executing this runbook should be logged in as a `foundry`-group member with
`sudo` access on `foundry-workspace`. Commits generated by the deploy (if any) use the
`ps-administrator` identity per workspace `CLAUDE.md` §8.

---

## Prerequisites

1. **Workspace VM access** — SSH into `foundry-workspace` as a `foundry`-group user.
2. **Tier A running** — `systemctl status local-slm.service` must show `active (running)`.
   The Doorman's systemd unit declares `Requires=local-slm.service`; if Tier A is not
   running, the Doorman unit will refuse to start.
3. **Cluster clone present** — `/srv/foundry/clones/project-slm/` exists and is on
   branch `cluster/project-slm`. The cluster clone is the source for both the pre-built
   binary and the env example.
4. **Rust toolchain** (only if building on-VM; skip if using the pre-built binary) —
   `cargo --version` should return the stable toolchain declared in
   `service-slm/rust-toolchain.toml`.

## Step 1: Build the binary (or skip if using pre-built)

### Option A — use the pre-built binary from the cluster clone

The release binary was built on the workspace VM as part of iter-19 and lives at:

```
/srv/foundry/clones/project-slm/service-slm/target/release/slm-doorman-server
```

Binary size: approximately 7.5 MB (stripped optimised release build).

Verify it is present and executable:

```bash
ls -la /srv/foundry/clones/project-slm/service-slm/target/release/slm-doorman-server
file /srv/foundry/clones/project-slm/service-slm/target/release/slm-doorman-server
```

Note: `target/` is gitignored. If the cluster clone's target directory has been cleaned
(e.g., after a `cargo clean`), proceed to Option B.

### Option B — build on the VM

```bash
cd /srv/foundry/clones/project-slm/service-slm
cargo build --release -p slm-doorman-server
```

Build time: approximately 2-3 minutes cold, 30 seconds incremental.
Output: `service-slm/target/release/slm-doorman-server`.

---

## Step 2: Place the binary

Install the binary to `/usr/local/bin/` with world-executable permissions:

```bash
sudo install -m 0755 \
    /srv/foundry/clones/project-slm/service-slm/target/release/slm-doorman-server \
    /usr/local/bin/slm-doorman-server
```

Verify:

```bash
/usr/local/bin/slm-doorman-server --version 2>/dev/null || \
    echo "(no --version flag; binary is present and executable)"
ls -la /usr/local/bin/slm-doorman-server
```

The binary does not expose a `--version` CLI flag — it starts as an HTTP server immediately.
A non-zero exit code from the above confirm the binary is present; the message is expected.

---

## Step 3: Place the env file

Create the directory and install the env file:

```bash
sudo mkdir -p /etc/local-doorman
sudo install -m 0640 \
    /srv/foundry/clones/project-slm/service-slm/docs/deploy/local-doorman.env.example \
    /etc/local-doorman/local-doorman.env
sudo chown root:local-doorman /etc/local-doorman/local-doorman.env
```

Review the installed file and confirm the values match the workspace dogfood configuration:

```bash
sudo cat /etc/local-doorman/local-doorman.env
```

Key values to verify:

| Variable | Expected value |
|---|---|
| `SLM_BIND_ADDR` | `127.0.0.1:9080` |
| `SLM_LOCAL_ENDPOINT` | `http://127.0.0.1:8080` |
| `SLM_APPRENTICESHIP_ENABLED` | `true` |
| `SLM_AUDIT_DIR` | `/var/lib/local-doorman/audit/` |
| `FOUNDRY_ROOT` | `/srv/foundry` |

The `SLM_TIER_C_*` block is commented out by default. Do not uncomment until the Anthropic
API key is ready to supply — leaving the block commented means `audit_proxy` returns 503
with a clear "unconfigured" message, which is the correct and expected behaviour.

## Step 4: Create the audit-ledger directory

The audit ledger is append-only JSONL, written by the `local-doorman` service user.
The directory must exist and be writable by that user before the server starts.

```bash
sudo mkdir -p /var/lib/local-doorman/audit
sudo chown -R local-doorman:local-doorman /var/lib/local-doorman
sudo chmod 750 /var/lib/local-doorman
sudo chmod 750 /var/lib/local-doorman/audit
```

Verify:

```bash
ls -la /var/lib/local-doorman/
```

The `local-doorman` service user and group are created by the existing unit install. If they
do not exist, create them before this step:

```bash
sudo useradd --system --no-create-home \
    --home-dir /var/lib/local-doorman \
    --shell /usr/sbin/nologin \
    local-doorman
```

## Step 5: Update or install the systemd unit

The baseline systemd unit lives at:

```
/srv/foundry/infrastructure/local-doorman/local-doorman.service
```

**If the unit is already installed** (check with `systemctl status local-doorman.service`):

Add an `EnvironmentFile=` directive via a systemd drop-in override so the env file from
Step 3 takes precedence over the inline `Environment=` lines in the base unit:

```bash
sudo mkdir -p /etc/systemd/system/local-doorman.service.d
sudo tee /etc/systemd/system/local-doorman.service.d/env-file.conf <<'EOF'
[Service]
EnvironmentFile=/etc/local-doorman/local-doorman.env
EOF
```

**If the unit is not yet installed** (fresh deploy):

```bash
sudo cp /srv/foundry/infrastructure/local-doorman/local-doorman.service \
    /etc/systemd/system/local-doorman.service

# Then add the env-file drop-in as above.
sudo mkdir -p /etc/systemd/system/local-doorman.service.d
sudo tee /etc/systemd/system/local-doorman.service.d/env-file.conf <<'EOF'
[Service]
EnvironmentFile=/etc/local-doorman/local-doorman.env
EOF
```

The base unit already sets `SLM_APPRENTICESHIP_ENABLED=true` and sensible defaults.
The env file from Step 3 is the operator-managed layer; drop-in takes precedence.

## Step 6: Reload and restart

```bash
sudo systemctl daemon-reload
sudo systemctl restart local-doorman.service
```

Check status:

```bash
sudo systemctl status local-doorman.service
```

Expected output includes:

```
● local-doorman.service — service-slm Doorman (three-tier inference router + audit ledger)
   Active: active (running) since ...
...
... service-slm Doorman starting ... has_local=true has_yoyo=false has_external=false apprenticeship_enabled=true
```

If `has_local=false`, the Tier A endpoint is unreachable — check `local-slm.service` and
confirm `SLM_LOCAL_ENDPOINT` is correct.

To follow live logs:

```bash
journalctl -fu local-doorman.service
```

## Step 7: Smoke-test

Run the smoke-test script from the cluster clone:

```bash
DOORMAN_URL=http://127.0.0.1:9080 \
    /srv/foundry/clones/project-slm/service-slm/scripts/smoke-test-doorman.sh
```

Expected output: all tests report PASS except the `audit/proxy` unconfigured-503 test,
which expects and reports PASS on a 503 response (503 is the correct behaviour when Tier C
is not configured).

The `POST /v1/chat/completions` test will PASS when `local-slm.service` is running and
responding. If Tier A is not reachable (cold-start or transient), it reports the error
without blocking the script — smoke-test mode is advisory.

Full expected summary line: `8 tests passed / 8 total` (or similar depending on script
version).

## Step 8: Verify the flow

Confirm the apprenticeship corpus and audit ledger are being written.

### Audit ledger

After at least one request through the Doorman, check the ledger:

```bash
ls -lt /var/lib/local-doorman/audit/
# Expected: one .jsonl file per day, named YYYY-MM-DD.jsonl
sudo tail -1 /var/lib/local-doorman/audit/$(date +%Y-%m-%d).jsonl | python3 -m json.tool
```

Expected fields in a Tier A audit entry: `request_id`, `module_id`, `tier`, `model`,
`inference_ms`, `cost_usd`, `completion_status`.

### Engineering corpus

The `capture-edit` hook writes engineering corpus tuples at every commit. Run the
corpus-stats script:

```bash
/srv/foundry/clones/project-slm/service-slm/scripts/corpus-stats.sh
```

Expected output: tuple count, date range, and schema confirmation for recent events.
Once `SLM_APPRENTICESHIP_ENABLED=true` is live, apprenticeship corpus tuples also appear
under `/srv/foundry/data/training-corpus/apprenticeship/`.

## Rollback procedure

If the new binary causes issues, roll back to the previous version:

1. Stop the service: `sudo systemctl stop local-doorman.service`.
2. Replace the binary with the previous version (if a backup exists at
   `/usr/local/bin/slm-doorman-server.prev`, restore it).
3. Restart: `sudo systemctl start local-doorman.service`.

If no previous binary backup exists, rebuild from the last known-good commit on the
`cluster/project-slm` branch:

```bash
cd /srv/foundry/clones/project-slm
git log --oneline -5          # identify the last known-good commit
git checkout <commit>  -- service-slm/  # (do not do this — rebuild instead)
# Correct approach: cargo build --release from the last known-good checkout
```

To revert the unit to its baseline (no env-file drop-in):

```bash
sudo rm /etc/systemd/system/local-doorman.service.d/env-file.conf
sudo systemctl daemon-reload
sudo systemctl restart local-doorman.service
```

## Troubleshooting

### Port already in use

```
failed to bind 127.0.0.1:9080: Address already in use (os error 98)
```

Another process has port 9080. Find it: `sudo ss -tlnp | grep 9080`. If it is a previous
Doorman instance, stop it: `sudo systemctl stop local-doorman.service` and check for stale
processes with `pgrep slm-doorman-server`.

### Audit-ledger directory missing or not writable

```
SLM_AUDIT_DIR create_dir_all failed; falling back to default
```

The `SLM_AUDIT_DIR` path either does not exist or the `local-doorman` user cannot write to
it. Run Step 4 again. Check ownership: `ls -la /var/lib/local-doorman/`.

### Tier A unreachable

```
[readyz] has_local=true, has_yoyo=false ...
[POST /v1/chat/completions] ...upstream error...
```

`local-slm.service` is not running or is not listening on `127.0.0.1:8080`. Check:
`systemctl status local-slm.service` and `curl -s http://127.0.0.1:8080/health`.

### HOME unset

```
failed to open audit ledger; ensure HOME is set
```

The systemd unit must set `Environment="HOME=/var/lib/local-doorman"`. This is present
in the baseline unit. If the env-file drop-in overwrites HOME inadvertently, add it back.

### Apprenticeship endpoints return 404

`SLM_APPRENTICESHIP_ENABLED` is not set to `true` or `1`. Check the env file:
`sudo grep APPRENTICESHIP /etc/local-doorman/local-doorman.env`.
