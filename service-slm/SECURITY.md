# Security Policy

## Reporting a vulnerability

**Do not open a public GitHub issue for security-sensitive reports.**

Email the maintainers directly at **peter@woodfinegroup.com**. Put `[security]`
in the subject line so the report surfaces immediately. If you want to encrypt
the report, request a PGP key in your first message and we will reply with
one.

What to include:

- A clear description of the vulnerability, including the affected version(s)
  or commit SHA.
- Reproduction steps or a minimal proof-of-concept, if you have one.
- Your assessment of severity and impact.
- Your preferred credit attribution (name, handle, or anonymous).

We aim to acknowledge receipt within 72 hours and to provide a substantive
response — including a remediation timeline — within 14 days.

If you do not receive an acknowledgement within 72 hours, assume the email
did not reach us and re-send with `[security URGENT]` in the subject.

## Supported versions

Until we ship a 1.0.0 release, only the latest tagged release and the `main`
branch are eligible for security fixes. Pre-1.0.0 versions are not considered
stable and may change without notice. Post-1.0.0 a supported-versions table
will appear here.

## Disclosure timeline

Our default disclosure window is 90 days from the date of the first
acknowledged report. If a reporter prefers a shorter or longer window, we
negotiate before publishing. In exceptional cases (e.g. the bug is being
actively exploited), we may publish sooner. In all cases the reporter is
credited in the advisory unless they request otherwise.

## Verifying releases

Every published release is signed via [Sigstore](https://www.sigstore.dev/)
using keyless signing with an OIDC identity. To verify a release artefact:

```bash
# Install cosign (https://docs.sigstore.dev/cosign/installation/)
cosign verify-blob \
  --bundle service-slm-vX.Y.Z.bundle \
  --certificate-identity-regexp 'https://github.com/woodfinegroup/.+' \
  --certificate-oidc-issuer 'https://token.actions.githubusercontent.com' \
  service-slm-vX.Y.Z.tar.gz
```

A convenience wrapper is at [`scripts/verify-sigstore.sh`](./scripts/verify-sigstore.sh).

Each release also ships:

- **SBOM** (CycloneDX JSON) generated by `cargo sbom`, named
  `service-slm-vX.Y.Z.sbom.json`.
- **Third-party notices** (Markdown) generated by `cargo about`, named
  `THIRD-PARTY-NOTICES.md`.
- **SLSA provenance attestation** produced by the release workflow.

If any of these are missing from a release you have downloaded, do not
install it. Verify from source instead and report the anomaly to
peter@woodfinegroup.com.

## Dependency discipline

We do not accept dependencies under copyleft licences. The allow-list is in
[`deny.toml`](./deny.toml) and is enforced by `cargo deny check licenses` on
every commit in CI. `cargo audit` also runs on every commit to catch known
CVEs in transitive dependencies.

If you discover a dependency has been compromised or carries an undisclosed
licence, treat it as a security vulnerability and use the disclosure process
above.

## Scope of this policy

In scope:

- The source code in this repository.
- The signed release artefacts distributed via GitHub Releases.
- The container images distributed via GitHub Container Registry (when
  published).

Out of scope:

- Third-party dependencies. If you find a vulnerability in a dependency,
  report it to the upstream project first. If the upstream project is
  unresponsive, mail us and we will decide whether to coordinate.
- Hypothetical misuse of the service (for example, deploying `service-slm`
  in a network configuration that exposes the doorman endpoint to the
  public internet). Misconfiguration is an operator concern; undocumented
  insecure-by-default behaviour is ours.
- Social-engineering or physical-security concerns.
